## 治理原则
- 组织层定义底线，不定义技术栈细节
- 技术栈层可以更严格，但不能更宽松
- 每条规则必须可被 AI 在特定环节执行
- 无法执行的规则不进入正式约束
- 具体检测模式（正则、AST、工具规则）由技术栈层定义；组织层仅声明规则意图与严重级别

## 严重级别
- block: AI 必须遵守，违反时停止执行并告知用户原因
- warn: AI 应遵守，违反时提醒用户但不阻止执行

## 数据分级
- public: 公开信息，无限制
- internal: 内部信息，API 响应需脱敏 → SEC-004
- sensitive: 敏感业务数据，禁止出现在日志中，禁止真实数据做测试 → SEC-001, SEC-004, SEC-007, TEST-001
- secret: 核心密钥，禁止出现在任何代码和 artifact 中，必须通过密钥管理服务获取 → SEC-001, SEC-004, SEC-007, TEST-001

## AI 行为底线
- 实现前说明对需求的理解和假设；有歧义时列出所有解读让用户选择 (block) [AI-001]
- 发现更简单方案时提出异议而非盲目执行 (warn) [AI-002]
- 不确定时停下来查阅或询问，不得猜测或虚构 (block) [AI-003]
- 只实现明确要求的功能，不引入未经要求的新依赖 (block) [AI-004]
- 不为一次性场景创建抽象层，复杂度过高时简化 (warn) [AI-005]
- 只碰必须碰的代码，不改动无关内容 (block) [AI-006]
- 每行修改须追溯到原始请求 (block) [AI-007]
- 任务转化为可验证成功标准，多步骤先列计划 (block) [AI-008]
- 编码后必须验证并报告结果；修复 bug 须先写重现测试 (block) [AI-009]
- 高风险变更必须人工最终批准后方可合并或上线 (block) [AI-010]
- 不得重构没坏的东西，即使认为有更好的写法 (warn) [AI-011]
- 不为不太可能发生的场景写错误处理 (warn) [AI-012]
- 代码量超过必要复杂度时必须简化，能 50 行解决的不要写 200 行 (warn) [AI-013]
- 自己引入的无用 import/变量/函数必须清理；预先存在的死代码只提示不删除 (warn) [AI-014]

## 语言规范
- artifact 使用正式书面中文撰写，技术术语可保留英文（如 JWT（JSON Web Token）） (block) [LANG-001]
- artifact 禁止口语化表达，中英文混排须加空格 (warn) [LANG-002]
- 代码注释使用中文，标识符使用英文 (block) [LANG-003]

## 命名规范
- change 命名格式：{type}/{product}/{desc}，类型限 feature/bugfix/refactor/enhancement，使用英文 (block) [NAM-001]
- 分支名必须与 change 名一致 (block) [NAM-002]
- artifact 文件名遵循约定：proposal.md / specs/*.md / design.md / tasks.md (block) [NAM-003]

## Proposal 规则
- 中文撰写，聚焦需求范围，不写技术实现 [ORG-P-001]
- 影响分析含：受影响系统、数据变更、接口变更 [ORG-P-002]
- 涉及敏感数据/风险/迁移/权限时标注影响和回退 [ORG-P-003]
- 接口字段变更时业务层面标注范围，技术细节留 design/specs [ORG-P-004]
- 能力描述可映射为后续独立 spec [ORG-P-005]
- 涉及 sensitive/secret 数据的变更必须标注数据级别 [ORG-P-006]

## 依赖管理
- 禁止引入未经核实的第三方依赖（需确认许可证兼容、维护状态、安全漏洞） (block) [DEP-001]
- 新增依赖必须在 design 中说明理由和替代方案评估 (block) [DEP-002]
- 依赖升级必须走变更流程，大版本升级需独立 change (warn) [DEP-003]

## Artifact 生成顺序
顺序：proposal → specs → design → tasks
追溯：specs→proposal；design→proposal+specs；tasks→design
约束：每个 artifact 只能追溯到先于它生成的 artifact（spec 不得引用尚未生成的 design）

## Specs 规则
- 每个 spec 聚焦单一行为，追溯到 proposal 能力点 [ORG-S-001]
- 含前置条件、输入、预期行为、异常场景、测试场景 [ORG-S-002]
- 不含实现代码和 SQL，可含接口定义和返回值结构 [ORG-S-003]
- 正常+异常场景（异常≥1），数据处理标注安全级别 [ORG-S-004]

## 安全红线
1. 禁止硬编码密钥、密码、token、secret [SEC-001]
2. 禁止拼接 SQL，必须参数化查询 [SEC-002]
3. 禁止明文存储密码 [SEC-003]
4. 禁止日志输出 sensitive/secret 数据或使用不安全日志配置 [SEC-004]
5. 禁止动态执行、反序列化或注入机制处理不可信输入 [SEC-005]
6. 禁止关闭安全校验或使用不安全默认配置 [SEC-006]
7. 禁止不安全数据处理（URL 敏感参数、SSRF、路径穿越） [SEC-007]
8. 禁止缺失访问控制 [SEC-008]

## Design 规则
- 引用 proposal 需求范围与 specs 定义的行为，不重复业务描述 [ORG-D-001]
- 说明变更影响范围（模块、文件、接口） [ORG-D-002]
- 数据存储→保护方案；外部接口→错误处理方案 [ORG-D-003]
- 不过度设计，不违反技术栈约束，优先复用已有模式 [ORG-D-004]
- 接口字段变更时含完整接口设计 [ORG-D-005]
- 变更涉及上线时必须包含回滚方案 [ORG-D-006]
- 新增依赖必须说明理由和替代方案评估 [ORG-D-007]

## 数据库安全底线
- 数据库连接信息禁止硬编码，必须使用环境变量或配置中心；账号区分只读/读写权限，查询只用只读用户 [SEC-DB-001]
- DDL/DML 变更先落脚本（含目的、影响、回滚），不得由 AI 直接执行 [SEC-DB-002]
- 脚本落盘后等人工确认方可执行 [SEC-DB-003]
- 生产环境禁止开发直连写数据 [SEC-DB-004]
- 批量数据操作必须有事务保护 [SEC-DB-005]

## 依赖管理
- 禁止引入未经核实的第三方依赖（需确认许可证兼容、维护状态、安全漏洞） (block) [DEP-001]
- 新增依赖必须在 design 中说明理由和替代方案评估 (block) [DEP-002]
- 依赖升级必须走变更流程，大版本升级需独立 change (warn) [DEP-003]

## Tasks 规则
- 每个 task 有验收标准，追溯到 design 方案点 [ORG-T-001]
- 代码变更 task 含测试要求；接口字段变更含同步更新任务 [ORG-T-002]
- 粒度可独立完成验证，标注优先级和依赖关系 [ORG-T-003]

## 安全红线（不可违反）
1. 禁止硬编码密钥、密码、token、secret [SEC-001]
2. 禁止拼接 SQL，必须参数化查询 [SEC-002]
3. 禁止明文存储密码 [SEC-003]
4. 禁止日志输出 sensitive/secret 数据或使用不安全日志配置 [SEC-004]
5. 禁止动态执行、反序列化或注入机制处理不可信输入 [SEC-005]
6. 禁止关闭安全校验或使用不安全默认配置 [SEC-006]
7. 禁止不安全数据处理（URL 敏感参数、SSRF、路径穿越） [SEC-007]
8. 禁止缺失访问控制 [SEC-008]

## 数据库安全底线
- 数据库连接信息禁止硬编码，必须使用环境变量或配置中心；账号区分只读/读写权限，查询只用只读用户 [SEC-DB-001]

## 代码质量
- 错误必须明确上报，不得静默吞异常 (block) [CQ-001]
- 错误信息须具体且含可操作恢复建议 (warn) [CQ-002]
- 测试覆盖正常+异常路径；bugfix 含回归测试 (block) [CQ-003]
- 禁止空测试、skip 测试或仅验证不抛异常的测试 (warn) [CQ-004]
- 新增代码的单元测试行覆盖率必须超过 80% (block) [CQ-005]
- 禁止绕过类型检查（as any / @ts-ignore / @ts-expect-error / eslint-disable） (block) [CQ-006]
- 超时场景必须显式处理，不得依赖无限等待 (warn) [CQ-007]
- 测试描述须说明测试意图而非实现细节 (warn) [CQ-008]
- 涉及数据处理的功能必须测试边界值（空值、超长、特殊字符、null） (warn) [CQ-009]
- 不得生成占位实现（TODO/FIXME/HACK）作为最终交付 (warn) [NG-001]
- 不得过度注释或重复代码已表达的内容 (warn) [NG-002]

## 性能底线
- 禁止已知 N+1 查询模式进入交付 (block) [PERF-001]
- 批量操作必须分页或流式处理，禁止全量加载 (block) [PERF-002]
- 涉及外部调用的循环必须评估批量或缓存方案 (warn) [PERF-003]

## 可观测性
- 关键业务操作必须有结构化日志 (block) [OBS-001]
- 外部调用必须记录耗时和结果状态 (block) [OBS-002]
- 错误日志必须包含足够上下文用于定位（请求 ID、操作名、关键参数——脱敏后） (block) [OBS-003]

## DDL/DML 目录约定
- DDL 脚本存放于 db/migrations/，DML 脚本存放于 db/data-fixes/ (block) [CONV-001]
- DDL/DML 脚本文件命名：{date}_{brief_description}.sql (warn) [CONV-002]
- DDL/DML 脚本须包含：目的说明、影响范围评估、回滚方案 (block) [CONV-003]

## Apply 阶段规则
- 只碰必须碰的代码，匹配现有代码风格 [ORG-A-001]
- 每行修改须追溯到 tasks 中的具体 task [ORG-A-002]
- 编码后必须执行验证并报告结果 [ORG-A-003]
- 安全规则（SEC-*）全部适用，不可豁免 [ORG-A-004]
- 新增代码的单元测试行覆盖率必须超过 80% [CQ-005] [ORG-A-007]
- 性能规则（PERF-*）全部适用 [ORG-A-005]
- 可观测性规则（OBS-*）全部适用 [ORG-A-006]

## 代码质量（verify 相关）
- 测试覆盖正常+异常路径；bugfix 含回归测试 (block) [CQ-003]
- 禁止空测试、skip 测试或仅验证不抛异常的测试 (warn) [CQ-004]
- 禁止绕过类型检查（as any / @ts-ignore / @ts-expect-error / eslint-disable） (block) [CQ-006]

## 代码审查
- 编译/解析失败、确定错误结果、违反安全红线的代码必须标记 (block) [CR-001]
- 代码风格偏好、linter 可检测、无关预存问题不得标记为阻塞 (warn) [CR-002]
- 至少 1 名 reviewer 审批后方可合并 (block) [CR-003]
