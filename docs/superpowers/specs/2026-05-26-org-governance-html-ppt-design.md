# 组织级别规范 HTML PPT 设计文档

## 1. 目标

创建一份**单文件 HTML** 形式的 PPT，用于面向团队进行“组织级别规范”宣讲。

该 PPT 不追求逐条覆盖全部规则，而是要在 **8-10 分钟** 的宣讲场景中，帮助研发、测试、产品和相关协作角色快速理解：

- 为什么组织层需要统一规范
- 这套规范如何分层设计
- 规范包含哪些核心部分
- 规则如何被 AI、人工和平台共同执行
- 为什么它能形成治理闭环

## 2. 范围

### In Scope

- 基于 `org/org-merged-draft.yaml` 的内容制作演示稿
- 生成单文件 HTML，打开即可演示
- 现代科技风视觉风格
- 10 页左右的高层概览型内容结构
- 支持键盘左右翻页
- 包含简单过渡动画、页码和章节层次提示

### Out of Scope

- 不实现复杂演讲者模式
- 不依赖外部 CDN、外部字体或在线资源
- 不做逐条规则手册式展开
- 不包含技术栈层细则、语言/框架专项检测规则
- 不在本阶段生成多套主题皮肤

## 3. 受众与使用场景

### 受众

- 研发工程师
- 测试工程师
- 产品经理
- 技术负责人

### 使用场景

- 团队内部宣讲
- 组织级规范介绍会
- 新项目启动前的统一约束说明

## 4. 内容策略

采用 **“总览地图型”** 讲法，主线为：

1. 为什么需要组织级规范
2. 这套规范按什么原则设计
3. 规范体系包含哪些模块
4. 每个模块分别约束什么
5. 这套规范如何执行、如何闭环

内容呈现上遵循以下原则：

- **讲框架，不念文档**：强调结构、关系、边界，而非罗列全部规则
- **讲治理，不讲技术细节**：突出组织层底线，不下沉到技术栈实现
- **讲闭环，不讲口号**：明确 AI、人工、平台如何共同执行
- **讲地图，不讲碎片**：让听众形成完整心智模型

## 5. 页面结构

### 第 1 页：封面

标题：组织级别规范

副标题建议：

- 面向 AI 编码时代的统一治理框架

目的：建立正式、现代、科技化的第一印象。

### 第 2 页：为什么需要组织级规范

核心信息：

- AI 编码能力增强，但输出质量和行为边界不稳定
- 团队之间规则口径不一致，会导致交付标准漂移
- 安全、质量、流程要求若不统一，治理成本会持续上升
- 组织层规范的作用是定义所有项目都不能突破的底线

目的：说明规范存在的必要性。

### 第 3 页：四条设计原则

展示 `org-merged-draft.yaml` 中的元原则：

- 组织层定义底线，不定义技术栈细节
- 技术栈层可以更严格，但不能更宽松
- 每条规则必须可被 AI 在特定环节执行
- 无法执行的规则不进入正式约束

目的：说明这不是传统制度文档，而是一套可执行治理规范。

### 第 4 页：六大部分全景图

展示六大模块：

- AI 编码行为底线
- 安全红线
- 代码质量规范
- SDD 流程治理
- Artifact 生成规则
- 冲突与豁免机制

目的：给听众建立整体地图。

### 第 5 页：三条核心约束主线

聚焦前三个最核心的约束域：

- AI：要求思考、简化、精准改动、验证结果
- 安全：定义不可突破的安全红线
- 质量：要求错误处理、测试、审查和 No-Go 反模式控制

目的：说明“编码行为、风险控制、交付质量”是最基础的三条主线。

### 第 6 页：流程治理

展示流程阶段：

`proposal → specs → design → tasks → implementation`

强调：

- 每个阶段有 gate
- 每个阶段有 owner
- 不是直接写代码，而是先把目标、规格、设计和任务拆解清楚

目的：说明规范不仅管代码，也管研发过程。

### 第 7 页：Artifact 规则

强调四阶段 artifact 的职责边界：

- proposal 讲“为什么做、做什么”
- specs 讲“需求和行为定义”
- design 讲“如何实现”
- tasks 讲“如何落地执行”

同时突出：

- 每阶段有“不能写什么”的边界
- 有跨阶段追溯关系
- specs 与 tasks 中测试要求贯穿始终

目的：让团队理解文档不是形式，而是协作边界。

### 第 8 页：冲突与豁免机制

强调：

- 默认策略是 stricter wins
- SEC-* 和 NG-* 不可豁免
- AI-*、PL-*、CQ-*、ORG-* 等在审批条件下可豁免
- 定性冲突无法判断时，AI 必须停下来询问人工

目的：说明这套规范不是死板规则堆砌，而是有治理决策机制。

### 第 9 页：组织层 vs 技术栈层

展示职责分层：

- 组织层：定义原则、底线、严重级别、执行者、通用检测原则
- 技术栈层：定义具体检测模式、正则/AST、误报说明、语言框架专项规则

目的：解释为什么组织层文档保持通用，而不是塞满 Java/Vue/MyBatis 细节。

### 第 10 页：总结与推行建议

总结语建议：

- 组织级规范不是为了增加文档负担，而是为了让 AI 时代的软件交付更可控、更一致、更可审计。

推行建议：

- 先统一组织层底线
- 再补齐各技术栈层规则
- 最后接入 AI、平台和人工审批形成闭环

目的：收束全场，给出行动方向。

## 6. 视觉设计

### 风格

- 现代科技风
- 深色背景
- 蓝紫渐变 + 青色高亮
- 适度发光、网格、线框、数据卡片元素

### 视觉关键词

- governance
- framework
- control
- system
- execution

### 配色建议

- 背景主色：深蓝 / 深紫 / 蓝黑渐变
- 标题强调：高亮青色或亮蓝色
- 正文：浅灰白
- 安全/阻断提示：橙红色点缀

### 版式原则

- 每页一个核心结论
- 使用卡片和信息块分组信息
- 尽量减少整段长文字
- 通过编号、线条、箭头、矩阵强化结构感

## 7. 交互设计

HTML PPT 应支持：

- 左右方向键切页
- 页码显示
- 当前页高亮
- 简单淡入/位移动画
- 适配 16:9 屏幕展示

不实现：

- 外部依赖的全功能幻灯片框架
- 演讲备注视图
- 复杂缩略图导航

## 8. 技术方案

### 结构

- 单个 `html` 文件
- 内嵌 `style` 和 `script`
- 每页作为独立 section 或 slide 容器

### 组成模块

- 封面页模板
- 内容页模板
- 总览图/流程图样式模块
- 页码与导航模块
- 键盘交互模块

### 实现要求

- 不依赖外网资源
- 不依赖第三方 JS 框架
- 在内网环境下可直接双击打开使用
- 样式和脚本保持可维护，避免写成难以调整的大段耦合代码

## 9. 内容来源约束

内容必须以 `org/org-merged-draft.yaml` 为唯一规范来源，并重点提炼以下信息：

- 元原则
- 六大模块划分
- AI / SEC / CQ / PL / Artifact / Conflict 的角色与关系
- artifact_rules 中的阶段职责边界
- pipeline 中的 gate 思路
- conflict_policy 与 exemption 的治理逻辑

不得：

- 虚构规范外的新制度
- 添加技术栈层细节作为组织层规则
- 将宣讲稿写成逐条规则清单

## 10. 成功标准

- 单文件 HTML 可直接打开演示
- 视觉风格统一，符合现代科技风
- 10 页左右即可讲清规范全景
- 团队成员能在一次宣讲后理解：这套规范为何存在、包含什么、如何执行
- 内容重点是框架和关系，而不是细枝末节

## 11. 风险与应对

### 风险 1：内容过多，导致像“念制度”

应对：每页只保留一个核心结论，使用图示和结构化表达。

### 风险 2：页面太炫但信息不清楚

应对：视觉只做辅助，不牺牲可读性，控制装饰元素密度。

### 风险 3：下沉到技术细节，偏离组织层定位

应对：所有内容围绕组织层原则、模块边界和执行机制组织，不展示框架级细节。

### 风险 4：页面缺乏宣讲节奏

应对：按照“问题 → 原则 → 地图 → 主线 → 执行 → 总结”的叙事顺序设计。
